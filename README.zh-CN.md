# deep-trace

[English](README.md)

**AI Agent 根因分析协议 — 从丰田"5 Whys"改造而来的五问审查法**

当 AI agent 出错时，本能反应是修复症状 — 加一条规则，告诉它"别这样做"。这些补丁不断累积，指令越来越臃肿，而同一类错误换个名字再次出现。

**deep-trace 打破了这个循环。** 它是一套系统化的五问审查法，强制调查者将每个错误追溯到指令/流程/设计层的根因 — 而非停留在症状层。

---

## Before vs. After：deep-trace 为什么存在

一起真实生产事故。**Dispatcher agent 在 50 秒内向一个已经回复过的 Router 发送了 7 条完全相同的路由请求。**

### 🔴 没有 deep-trace（常规做法）

| 步骤 | 操作 | 结果 |
|------|--------|---------|
| **看到症状** | "Dispatcher 在重复自己" | 归为"agent 行为异常" |
| **应用修复** | 加规则："不要发送重复路由请求" | 规则部署到 Dispatcher 指令中 |
| **验证** | 等待。接下来一小时内没有重复请求。 | ✅ 标记为已解决 |
| **现实** | 两周后，另一个 agent 把已完成部署重新执行了 5 次。同一个根因，不同的名字。 | 🔴 问题回来了 |

**为什么修复失败了**："不要发送重复路由请求"是一个症状补丁。它压制了一个 agent 上的一个症状。根因 — 无状态 agent 在每次触发事件时从零开始重新评估 — 完全没有被触及。它换个面具在其他地方出现。

### 🟢 有 deep-trace（五问根因审查法）

| 问题 | 发现 |
|----------|---------|
| **Q1：事实** | 时间线：50 秒内 7 条相同路由请求，Router 在第一条后就已回复 |
| **Q2：直接原因** | Dispatcher 指令缺少"路由前，检查该任务是否已经被路由过"。每次触发事件导致 Dispatcher 从零开始 — 没有跨运行的状态感知。 |
| **Q3：为什么存活** | 三个具体的流程缺口：(1) 路由工作流中没有去重检查步骤，(2) @mention 规则覆盖了防沉默但未覆盖防重复触发，(3) 没有规则"如果执行者说了'done'，不要再 @mention 它们" |
| **Q4：检测链** | 7 条请求的爆发 **12.75 小时**未被检测到。检测存在（人类审查）但没有实时组件。每个反馈回路都把每条请求当作独立的。 |
| **Q5：根因模式** | **原理**：当一个无状态 agent 被同一上下文中的事件重新触发，且没有跨运行的状态感知时，它会从零开始重新评估并可能重复执行已完成的动作。指令必须包含一个强制性的"行动前检查已有结果"步骤。 |

**替换测试**：将"无状态 agent" → "[无状态处理单元]"，"被重新触发"→"[被重新调用]" — 仍然成立。这是一个**通用原理**，不是 Dispatcher 专属补丁。

**部署了三项改进措施**：
1. **路由前去重检查** — 路由前检查，而非路由后
2. **已完成 agent 的 @mention 防护** — 使用 issue 链接，而非 agent mention
3. **突发检测监控**（待办） — 同一 agent 在 60 秒内发 ≥ 3 条近似相同评论时告警

**结果**：同一类错误**再也没有复发** — 跨任何 agent、任何触发类型、任何任务域。因为修复针对的是模式，而非症状。

### 一句话说清差异

| 没有 deep-trace | 有 deep-trace |
|---|---|
| "Dispatcher 不应该重复自己。" | "无状态 agent 需要强制性的行动前状态检查。" |
| 为**一个 agent**修复了**一个症状**。 | 为**所有 agent**修复了**一整类故障**。 |
| 问题两周后换个名字回来。 | 问题不再回来。 |

## 范围：deep-trace 是什么（不是什么）

deep-trace 是**事后根因分析** — 你在事件发生后运行它。它是更广泛质保体系中的一个组件：

```
                ┌──────────────────────────┐
                │  patrol-signal-detection  │  ← "什么时候该调查？"
                │  (实时 S1–S9 信号         │
                │   监控)                   │
                └────────────┬─────────────┘
                             │ 触发
                             ▼
                ┌──────────────────────────┐
                │  deep-trace (本仓库)      │  ← "根因是什么？"
                │  (事后五问 RCA 审查法)    │
                └────────────┬─────────────┘
                             │ 产出改进项
                             ▼
                ┌──────────────────────────┐
                │  agent-pregating-check   │  ← "如何防止复发？"
                │  (输出前三问自检)         │
                └──────────────────────────┘
```

| 技能 | 角色 | 时机 |
|-------|------|------|
| **deep-trace**（本仓库） | 事后 RCA — 五问审查法 | 事件发生后 |
| agent-pregating-check（计划中） | 输出前闸门 — "产出前三问" | 每次回复前 |
| agent-review-standards（计划中） | 日常质量审查 — 4D 标准 | 常规质量检查 |
| patrol-signal-detection（计划中） | 实时 S1–S9 信号监控 | 持续运行 |

## deep-trace 的独特之处

市面上有很多"5 Whys"教程。deep-trace 不是又一个方法论科普 — 它是从 60+ 生产事件中提炼出的**AI agent 专用 RCA 协议**。

| 通用 5 Whys 教程 | deep-trace |
|---|---|
| "问五次为什么" | 每个问题都有**硬约束**和闸门检查（Q1：不能有因果词；Q2：必须指向具体行/操作；Q5：替换测试） |
| 没有 agent 专属概念 | **暂时性终止**（上下文截断 → 重跑 → 消失 → 停止）、**8 种错误签名码**（U-UND、U-CTX、E-FMT、E-SKP、I-RED、D-INC、P-RTE、P-REP）覆盖 LLM-agent 特有的故障模式 |
| 以报告结尾 | 以**可执行改进项**结尾：`[action]` 前缀 + 验收标准 + error_signature 标签用于跨事件追踪 |
| 规则是临时添加的 | **元原则**：提炼原则而非补丁；一条全局规则 > N 条实例补丁 |
| 没有自我应用 | **递归自省**：协议要求调查者对自己的失败应用同样的五问 |

## 快速开始

### 安装为 Claude Code Skill

```bash
# 复制到你的 skills 目录
cp -r deep-trace ~/.claude/skills/deep-trace/
```

当你调用 `/deep-trace` 或检测到 agent 错误信号时，skill 会自动激活。

### 嵌入 Agent 指令（精简版，约 15 行）

```
## 根因分析（五问协议）

分析错误时，按顺序遵循以下五个问题：

Q1：事实 — 仅时间线。不能有因果词（"因为"、"由于"）。
Q2：直接原因 — 定位到具体的指令/操作/决策。
    如果怀疑上下文截断：重跑 → 问题消失 → 标记为暂态，停止。
Q3：为什么存活 — 哪个确切的流程步骤未能捕获此错误？
Q4：为什么没发现 — 绘制完整检测链，标记断点，报告延迟。
Q5：根因模式 — 表达为通用原理。将所有专有名词替换为
    占位符。仍然成立 → 原理。崩塌 → 补丁，重写。

输出：根因 + 带可观测验收标准的改进项。
```

### 嵌入 Agent 指令（完整版，约 50 行）

参见 [SKILL.md](./SKILL.md#integration-with-existing-systems) 获取包含错误签名分类码的完整嵌入模板。

## 生产实践记录

在多 agent 系统中经过 60+ 次 agent 协作场景的开发与实战检验。关键成果：

- **消灭了重复路由爆发**（Case: Duplicate Routing Request Burst）：50 秒内 7 条相同请求 → 部署后零复发
- **终结了高频纠正级联**（Case: High-Frequency Correction Spiral）：40 分钟内 4 次纠正 → S7b 零复发
- **提炼了可复用的设计原则**（Case: Quality Standard Relativization）：模板绑定的规则不具备通用性 — 防止了整个未来错误类别
- **减少了指令补丁膨胀**：通过 RCA 添加的规则是原则，不是一次性症状禁令
